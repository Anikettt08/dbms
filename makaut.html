📋 SECTION A: 5-MARK QUESTIONS 5 Marks

Q1. What is DBMS? Explain its advantages over File System. ⭐

Answer: A Database Management System (DBMS) is software that manages databases, providing controlled access to data. It acts as an interface between users and the database.

Advantages over File System:

  • Data Redundancy: DBMS reduces duplicate data; file systems store redundant copies.
  • Data Inconsistency: Changes in one place reflect everywhere in DBMS; file systems have inconsistent updates.
  • Data Sharing: Multiple users can access data simultaneously in DBMS.
  • Data Security: DBMS provides authentication, authorization, and encryption; file systems have basic OS-level security only.
  • Data Integrity: DBMS enforces constraints (primary key, foreign key); file systems cannot.
  • Backup & Recovery: DBMS has built-in mechanisms; file systems are manual.

Q2. Explain Three-Schema Architecture (ANSI-SPARC) with diagram. ⭐

The three-schema architecture separates user applications from the physical database.

  1. External Level (View Level): The highest level, closest to users. Describes part of the database relevant to particular users. Different users have different views.
  2. Conceptual Level (Logical Level): Describes the structure of the entire database. Defines what data is stored, relationships, constraints, and semantic information. The DBA works at this level.
  3. Internal Level (Physical Level): Describes physical storage structure. Defines how data is actually stored on disk (indexes, file organization, compression).

Data Independence:

  • Logical Data Independence: Ability to change conceptual schema without affecting external schemas.
  • Physical Data Independence: Ability to change internal schema without affecting conceptual schema.

Q3. What is Data Dictionary? Explain its importance. ⭐

A Data Dictionary (or catalog) is a centralized repository of information about data. It stores metadata — data about data.

Contents:

  • Names of tables, attributes, and their data types
  • Constraints (primary key, foreign key, check constraints)
  • Relationships between tables
  • User information and access privileges
  • Index names and definitions

Importance: Helps DBMS manage data efficiently, ensures data integrity and consistency, assists in query optimization, and is useful for database design and documentation.

Q4. Explain ER Model with symbols. Draw ER Diagram for Library Management System. ⭐

The Entity-Relationship (ER) Model is a high-level conceptual data model used to describe the structure of a database.

Symbols:

  • Rectangle: Entity (e.g., Student, Book)
  • Ellipse/Oval: Attribute (e.g., Name, Roll_No)
  • Diamond: Relationship (e.g., Issues, Borrows)
  • Double Ellipse: Multivalued Attribute
  • Dashed Ellipse: Derived Attribute
  • Double Rectangle: Weak Entity

Library Management System ER Diagram:

  • Entities: Student, Book, Librarian
  • Attributes: Student (Roll_No, Name, Address), Book (ISBN, Title, Author, Price)
  • Relationship: "Issues" (Student issues Book)
  • Cardinality: One Student can issue many Books; One Book can be issued by one Student at a time (1:N)

Q6. What is Normalization? Explain 1NF, 2NF, and 3NF with examples. ⭐

Normalization is the process of organizing data to reduce redundancy and eliminate anomalies (insertion, update, deletion).

1NF (First Normal Form): All attributes contain only atomic (indivisible) values. No repeating groups. Example: A table with Student_Hobbies = "Cricket, Football" violates 1NF. Split into separate rows or a new table.

2NF (Second Normal Form): Must be in 1NF. No partial dependency (non-prime attributes must depend on the whole primary key, not just part of it). Example: In (Student_ID, Course_ID) → Student_Name, Course_Name. Student_Name depends only on Student_ID (partial dependency). Split into Student and Course tables.

3NF (Third Normal Form): Must be in 2NF. No transitive dependency (non-prime attributes must not depend on other non-prime attributes). Example: Roll_No → Dept_ID → Dept_Name. Dept_Name transitively depends on Roll_No. Split into Student and Department tables.

Q7. Explain ACID Properties of Transaction. ⭐

A transaction is a logical unit of work. ACID properties ensure reliability:

  • Atomicity: "All or nothing." A transaction is either fully completed or fully rolled back. If any part fails, the entire transaction is undone.
  • Consistency: A transaction brings the database from one valid state to another. All integrity constraints must be satisfied.
  • Isolation: Concurrent transactions execute independently. The intermediate state of one transaction is invisible to others.
  • Durability: Once committed, changes are permanent and survive system failures (stored in non-volatile memory).

Q8. Explain 2-Phase Locking Protocol. ⭐

2-Phase Locking (2PL) ensures serializability by dividing a transaction into two phases:

Phase 1: Growing Phase — The transaction acquires locks (shared or exclusive) as needed. No locks are released during this phase. The transaction continues to request new locks.

Phase 2: Shrinking Phase — Once the transaction releases its first lock, it enters the shrinking phase. No new locks can be acquired after this point. Locks are gradually released.

Types:

  • Basic 2PL: May have cascading rollback.
  • Strict 2PL: Exclusive locks held until commit/abort.
  • Rigorous 2PL: All locks held until commit/abort.

Q9. What is Concurrency Control? Explain Lost Update, Dirty Read, and Unrepeatable Read. ⭐

Concurrency Control manages simultaneous transactions to ensure database consistency.

Problems:

  1. Lost Update: Two transactions read the same data, modify it, and write back. The second write overwrites the first, losing the first update. Example: T1 and T2 both read A=100. T1 adds 10 (A=110), T2 subtracts 10 (A=90). T1's update is lost.
  2. Dirty Read (Temporary Update): A transaction reads data modified by another uncommitted transaction. If the other transaction rolls back, the read data is invalid. Example: T1 changes A=100 to A=110. T2 reads A=110. T1 rolls back. T2 has dirty data.
  3. Unrepeatable Read: A transaction reads the same data twice and gets different values because another transaction modified it in between.

📋 SECTION B: 10-MARK QUESTIONS 10 Marks

Q1. Explain Relational Algebra operations: Selection, Projection, Union, Set Difference, Cartesian Product, and Join. ⭐

Relational Algebra is a procedural query language.

  1. Selection (σ): Selects rows (tuples) that satisfy a condition. Syntax: σ_condition(R). Example: σ_Salary>50000(Employee) — selects employees with salary > 50,000.
  2. Projection (π): Selects specific columns (attributes). Syntax: π_attribute_list(R). Example: π_Name, Salary(Employee) — shows only Name and Salary columns.
  3. Union (∪): Combines two union-compatible relations, removing duplicates. Syntax: R ∪ S. Condition: Same number of attributes, compatible domains.
  4. Set Difference (−): Returns tuples in R but not in S. Syntax: R − S.
  5. Cartesian Product (×): Combines every tuple of R with every tuple of S. Syntax: R × S. Result: If R has m tuples and S has n tuples, result has m×n tuples.
  6. Join (⋈): Combines related tuples from two relations based on a condition. Natural Join: Equi-join on common attributes. Equi-Join: Join with equality condition. Theta Join: Join with general condition.

Q2. Explain SQL commands: DDL, DML, DCL, and TCL with examples. ⭐

DDL (Data Definition Language): Defines database structure.

CREATE TABLE Student (Roll INT PRIMARY KEY, Name VARCHAR(50));
ALTER TABLE Student ADD Age INT;
DROP TABLE Student;

DML (Data Manipulation Language): Manipulates data.

INSERT INTO Student VALUES (1, 'Rahul', 20);
UPDATE Student SET Age=21 WHERE Roll=1;
DELETE FROM Student WHERE Roll=1;
SELECT * FROM Student WHERE Age>18;

DCL (Data Control Language): Controls access.

GRANT SELECT ON Student TO User1;
REVOKE INSERT ON Student FROM User1;

TCL (Transaction Control Language): Manages transactions.

COMMIT;          -- saves changes permanently
ROLLBACK;        -- undoes changes
SAVEPOINT sp1;   -- creates a savepoint within a transaction

Q3. Write SQL queries for the following scenarios (Employee-Department schema). ⭐

Schema: Employee(Emp_ID, Name, Salary, Dept_ID), Department(Dept_ID, Dept_Name, Location)

1. Find all employees with salary > 50,000:

SELECT * FROM Employee WHERE Salary > 50000;

2. Find employee names and their department names:

SELECT E.Name, D.Dept_Name 
FROM Employee E, Department D 
WHERE E.Dept_ID = D.Dept_ID;

3. Find the highest salary in each department:

SELECT Dept_ID, MAX(Salary) 
FROM Employee 
GROUP BY Dept_ID;

4. Find departments with more than 5 employees:

SELECT Dept_ID, COUNT(*) 
FROM Employee 
GROUP BY Dept_ID 
HAVING COUNT(*) > 5;

5. Find the second highest salary:

SELECT MAX(Salary) 
FROM Employee 
WHERE Salary < (SELECT MAX(Salary) FROM Employee);

Q4. Explain BCNF with example. Why is it stricter than 3NF? ⭐

Boyce-Codd Normal Form (BCNF) is a stricter version of 3NF.

Definition: A relation is in BCNF if for every functional dependency X → Y, X must be a superkey.

Difference from 3NF: In 3NF, if X → Y and Y is a prime attribute (part of any candidate key), the dependency is allowed even if X is not a superkey. In BCNF, X must ALWAYS be a superkey for every non-trivial FD.

Example (3NF but not BCNF): Student(Student_ID, Subject, Professor)

  • FDs: Student_ID, Subject → Professor; Professor → Subject
  • Candidate Keys: {Student_ID, Subject} and {Student_ID, Professor}
  • Professor → Subject violates BCNF because Professor is not a superkey.
  • It is in 3NF because Subject is a prime attribute.

Decomposition to BCNF: R1(Professor, Subject), R2(Student_ID, Professor)

Why BCNF is better: Eliminates all anomalies caused by non-superkey determinants.

Q5. Explain Transaction States and Log-Based Recovery with Checkpoints. ⭐

Transaction States:

  1. Active: Transaction is executing.
  2. Partially Committed: Last statement executed, waiting for commit.
  3. Committed: Changes permanently saved.
  4. Failed: Normal execution cannot continue.
  5. Aborted: Transaction rolled back, database restored to previous state.

Log-Based Recovery: A log file records all database modifications:

  • [Start, T]: Transaction T started.
  • [Write, T, X, old_value, new_value]: T updated X.
  • [Commit, T]: T committed.
  • [Abort, T]: T aborted.

Checkpoints: A checkpoint is a point where all committed transactions are written to disk. During recovery, only transactions after the last checkpoint need to be examined.

  • Redo List: Transactions committed after checkpoint (must redo).
  • Undo List: Transactions active at checkpoint but not committed (must undo).

Recovery Process:

  1. Scan log backward from end to checkpoint.
  2. Identify committed and uncommitted transactions.
  3. Redo committed transactions.
  4. Undo uncommitted transactions.

Q1. Design a complete Database for a University System. Draw ER Diagram, convert to Relational Schema, and normalize to 3NF. ⭐

University Database ER Design:

Entities & Attributes:

  • Student: Student_ID (PK), Name, DOB, Email, Address
  • Course: Course_ID (PK), Course_Name, Credits
  • Professor: Prof_ID (PK), Name, Department
  • Department: Dept_ID (PK), Dept_Name

Relationships:

  • Enrolls: Student enrolls in Course (M:N)
  • Teaches: Professor teaches Course (1:N)
  • Belongs_To: Student belongs to Department (N:1)

Relational Schema Conversion:

  1. Student (Student_ID, Name, DOB, Email, Address, Dept_ID)
    PK: Student_ID | FK: Dept_ID references Department
  2. Course (Course_ID, Course_Name, Credits, Prof_ID)
    PK: Course_ID | FK: Prof_ID references Professor
  3. Professor (Prof_ID, Name, Department)
    PK: Prof_ID
  4. Department (Dept_ID, Dept_Name)
    PK: Dept_ID
  5. Enrolls (Student_ID, Course_ID, Grade)
    PK: (Student_ID, Course_ID) | FK: Student_ID references Student, Course_ID references Course

Normalization:

  • All relations are in 1NF (atomic values).
  • All relations are in 2NF (no partial dependencies; all non-key attributes depend on full PK).
  • All relations are in 3NF (no transitive dependencies; no non-key attribute depends on another non-key attribute).

Q2. Explain various types of Joins in SQL with examples. ⭐

1. INNER JOIN: Returns only matching rows from both tables.

SELECT * FROM Employee E 
INNER JOIN Department D ON E.Dept_ID = D.Dept_ID;

2. LEFT OUTER JOIN: Returns all rows from left table, matching rows from right, NULL for non-matching.

SELECT * FROM Employee E 
LEFT JOIN Department D ON E.Dept_ID = D.Dept_ID;

3. RIGHT OUTER JOIN: Returns all rows from right table, matching rows from left, NULL for non-matching.

SELECT * FROM Employee E 
RIGHT JOIN Department D ON E.Dept_ID = D.Dept_ID;

4. FULL OUTER JOIN: Returns all rows when match in either table, NULL where no match.

SELECT * FROM Employee E 
FULL OUTER JOIN Department D ON E.Dept_ID = D.Dept_ID;

5. CROSS JOIN: Cartesian product of both tables.

SELECT * FROM Employee CROSS JOIN Department;

6. SELF JOIN: Joins a table with itself.

SELECT E1.Name AS Employee, E2.Name AS Manager 
FROM Employee E1, Employee E2 
WHERE E1.Manager_ID = E2.Emp_ID;

Q3. Explain Deadlock in detail. How is it detected and prevented? ⭐

Deadlock: A situation where two or more transactions wait indefinitely for each other to release locks.

Necessary Conditions:

  1. Mutual Exclusion
  2. Hold and Wait
  3. No Preemption
  4. Circular Wait

Deadlock Detection:

  • Wait-For Graph (WFG): Nodes are transactions; edges represent waiting. A cycle indicates deadlock.
  • Timeout: If a transaction waits too long, assume deadlock.

Deadlock Prevention Strategies:

  1. Wait-Die (Non-Preemptive): Older transaction waits for younger; younger dies (rolls back) if it requests resource held by older.
  2. Wound-Wait (Preemptive): Older transaction "wounds" (aborts) younger; younger waits for older.
  3. Graph-Based Protocol: Lock data items in a specific order (tree protocol).
  4. Time-Stamp Ordering: Assign timestamps; transactions ordered by time to prevent cycles.

Deadlock Resolution: Choose a victim transaction (usually youngest or least work done). Roll back victim and release its locks. Restart victim later.

Q4. Explain Indexing in detail: Primary, Secondary, Clustering, and Multilevel Index. ⭐

Indexing: A data structure technique to improve search speed by creating pointers to data.

1. Primary Index: Built on ordering key field (primary key). Data file is sorted on the key. One index entry per data block. Dense or sparse.

2. Clustering Index: Built on ordering non-key field. Data file is sorted, but multiple records can have same value. One entry per distinct value.

3. Secondary Index: Built on non-ordering fields. Can be on key or non-key fields. Data file does not need to be sorted. More index entries than primary index.

4. Multilevel Index: When primary index is too large, create an index on the index. Outer index points to inner index blocks. Reduces search time from linear to logarithmic.
